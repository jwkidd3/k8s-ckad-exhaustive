🏷️ Labels and Annotations

Labels — Identifying Metadata

  • Key/value pairs used by selectors
  • Drive Service routing, Deployment ownership, RBAC scoping
  • Should describe identity, not implementation
  • Example: env=prod, team=shiny, app=v1.2.4

Annotations — Non-Identifying Metadata

  • Same key/value shape, but not selectable
  • For tooling: contact info, build SHA, ingress hints
  • Example: contact=John Doe, commit=2d3mg3

CLI Cheat-sheet

# Add and remove labels
kubectl label pod frontend env=prod team=shiny
kubectl label pod backend env-          # remove

# Selectors
kubectl get pods -l env=prod
kubectl get pods -l 'team in (shiny,legacy)'
kubectl get pods --show-labels

# Annotations
kubectl annotate pod frontend \
  contact="John Doe" commit=2d3mg3
kubectl get pods -o yaml | grep -C 3 annotations:

⚠️ API Deprecation & Migration

Why APIs Change

  • Stable (v1) APIs get long-term guarantees
  • Beta (v1beta1, v2beta2) APIs can change or be removed
  • Major workloads transitioned from extensions/v1beta1 → apps/v1
  • Ingress moved networking.k8s.io/v1beta1 → networking.k8s.io/v1
  • HPA: autoscaling/v1 → v2beta1 → v2

How to Find Trouble

# See available API versions on the cluster
kubectl api-versions

# Get the GVR of an existing resource
kubectl explain deployment

# Run kubectl convert to migrate old YAML
kubectl convert -f deployment.yaml \
  --output-version apps/v1

When apps/v1 replaced beta versions, spec.selector.matchLabels became required. Old beta manifests will fail to apply without it.

❤️‍🩹 Pod Probes

Three Probe Types

  • readiness — should this Pod receive traffic right now?
  • liveness — is this Pod still healthy or should kubelet restart it?
  • startup — give a slow-starting app extra time before liveness kicks in

Probe Mechanisms

  • httpGet — HTTP GET to a path/port
  • tcpSocket — TCP connect attempt
  • exec — run a command, expect exit 0
  • grpc — for gRPC health protocol

Tuning Knobs

  • initialDelaySeconds — wait before first probe
  • periodSeconds — how often
  • timeoutSeconds — probe timeout
  • failureThreshold — consecutive fails before action
  • successThreshold — required successes after failure
readinessProbe:
  httpGet: {path: /, port: nodejs-port}
  initialDelaySeconds: 2
livenessProbe:
  httpGet: {path: /, port: nodejs-port}
  initialDelaySeconds: 5
  periodSeconds: 8

📊 Metrics + kubectl top

How Metrics Flow

kubelet (cAdvisor) │ ▼ metrics-server (kube-system) │ ▼ metrics.k8s.io API │ ▼ kubectl top / HPA
  • metrics-server polls every 15s by default
  • Metrics are not persisted — that's Prometheus' job
  • HPA scaling decisions read this same API

Common Commands

kubectl top nodes
kubectl top pods
kubectl top pods -n stress-test
kubectl top pods --sort-by=memory

# Sort and pick the heaviest
kubectl top pods -n stress-test \
  --sort-by=memory --no-headers | head -1

If kubectl top errors out, check that metrics-server is running: kubectl -n kube-system get deploy metrics-server.

🐛 Debugging Pods

Triage Order

  1. kubectl get pods — phase & restart count
  2. kubectl describe pod <name> — events at the bottom
  3. kubectl logs <name> (and -p for the previous container)
  4. kubectl exec -it <name> -- sh
  5. kubectl debug for distroless or stuck Pods

kubectl debug

# Attach a debug container that shares process namespace
kubectl debug -it <pod> \
  --image=busybox:1.36 \
  --share-processes \
  --copy-to=<pod>-debug

Why --share-processes? Without it you can only see the debug container's own processes. With it, you can ps aux and see the broken app.
