📦 Deployments: Wrapping Pods for Production

The Hierarchy

Deployment └── ReplicaSet └── Pod, Pod, Pod ...
  • A Deployment owns one or more ReplicaSets
  • A ReplicaSet keeps N identical Pods running
  • You almost never create a ReplicaSet directly
  • Editing a Deployment's spec.template creates a new ReplicaSet

Minimal Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  labels:
    tier: backend
spec:
  replicas: 3
  selector:
    matchLabels:
      app: v1
  template:
    metadata:
      labels:
        app: v1
    spec:
      containers:
      - name: nginx
        image: nginx:1.23.0

Selector ↔ labels must match exactly. apps/v1 requires the selector to be set.

🔄 Rolling Updates

Strategy Knobs

  • maxSurge — extra pods allowed above replicas during update
  • maxUnavailable — pods allowed missing during update
  • Defaults: 25% / 25%
  • Recreate strategy is the alternative — drop all, then re-create (downtime)

Imperative Update

kubectl set image deployment/nginx \
  nginx=nginx:1.23.4
kubectl rollout status deployment/nginx
kubectl rollout history deployment/nginx
kubectl annotate deployment/nginx \
  kubernetes.io/change-cause="Pick up patch"

Rollback

kubectl rollout undo deployment/nginx
kubectl rollout undo deployment/nginx --to-revision=1
kubectl rollout pause deployment/nginx
kubectl rollout resume deployment/nginx

Backward compatibility: Rolling updates assume v1 and v2 can serve traffic simultaneously. Database schema changes, breaking API changes, or stateful protocols may need blue-green instead.

🟦🟩 Blue-Green Deployment

How it Works

  • Two parallel Deployments: nginx-blue + nginx-green
  • Each Pod template carries a version=blue|green label
  • One Service selects one version at a time
  • Cutover = change the Service selector
  • Rollback = flip selector back; old Pods still running

When to Use It

  • Breaking changes that can't run side-by-side
  • Need instant rollback (no rollout time)
  • Willing to pay 2× capacity briefly

Selector Cutover

apiVersion: v1
kind: Service
metadata:
  name: nginx
spec:
  type: ClusterIP
  selector:
    version: blue   # ← flip to "green"
  ports:
  - port: 80
    targetPort: 80
kubectl patch service nginx \
  -p '{"spec":{"selector":{"version":"green"}}}'

📈 Horizontal Pod Autoscaler

HPA in One Sentence

Watches a metric (CPU, memory, or custom) and scales replicas of a Deployment up or down to keep the metric inside a target range.

Requirements

  • Metrics-server running in the cluster
  • Pods declare resources.requests — HPA computes utilization vs the request
  • Use autoscaling/v2 for multi-metric HPAs

Two-Metric HPA

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx
  minReplicas: 3
  maxReplicas: 8
  metrics:
  - type: Resource
    resource:
      name: cpu
      target: {type: Utilization, averageUtilization: 75}
  - type: Resource
    resource:
      name: memory
      target: {type: Utilization, averageUtilization: 60}

⎈ Building a Helm Chart

Chart.yaml (Helm v3)

apiVersion: v2     # required for Helm 3
name: web-app
version: 2.5.4     # chart semver

Pitfall: apiVersion is v2 for Helm 3 charts. Don't confuse it with chart version (2.5.4) or app version.

values.yaml

service_port: 80
container_port: 3000

A Template

apiVersion: v1
kind: Service
metadata:
  name: web-app-service
  namespace: {{ .Release.Namespace }}
spec:
  type: ClusterIP
  selector:
    app: hello-world
  ports:
  - protocol: TCP
    port: {{ .Values.service_port }}
    targetPort: {{ .Values.container_port }}
helm package .
helm install hello-world . \
  --namespace app-stack --create-namespace \
  --set service_port=9090
