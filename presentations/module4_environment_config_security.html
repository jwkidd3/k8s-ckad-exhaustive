🧬 CustomResourceDefinitions

Extending the API

  • A CRD defines a new resource kind
  • Cluster users can then create instances of that kind via kubectl, like any built-in
  • Operators add controllers that act on the resource
  • API path: /apis/<group>/<version>

Modern CRD apiVersion

Always use apiextensions.k8s.io/v1. The beta version was removed in Kubernetes 1.22.

apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: backups.example.com
spec:
  group: example.com
  scope: Namespaced
  names:
    kind: Backup
    singular: backup
    plural: backups
  versions:
  - name: v1
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            properties:
              cronExpression: {type: string}
              podName: {type: string}
              path: {type: string}

🛡️ RBAC + ServiceAccounts

Four Building Blocks

  • ServiceAccount — identity for Pods
  • Role / ClusterRole — set of allowed verbs on resources
  • RoleBinding — namespace-scoped attachment of a Role to a subject
  • ClusterRoleBinding — cluster-wide attachment

Subject Types

  • ServiceAccount (most common in-cluster)
  • User & Group (external identity)
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: list-services-clusterrole
rules:
- apiGroups: [""]
  resources: ["services"]
  verbs: ["list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: sa-services-rb
  namespace: t23
subjects:
- kind: ServiceAccount
  name: api-call
  namespace: t23
roleRef:
  kind: ClusterRole
  name: list-services-clusterrole
  apiGroup: rbac.authorization.k8s.io

  • ✅ ClusterRole to list services
  • ✅ RoleBinding tying SA → ClusterRole
  • ✅ Pod uses SA + curl to the API server with its token

⚖️ Resource Requests & Limits

Two Numbers per Resource

  • requests — what the scheduler reserves
  • limits — hard cap the runtime enforces
  • CPU: throttled if over limit
  • Memory: OOMKilled if over limit

Why It Matters

  • Without requests, scheduler can't pack nodes
  • Without limits, one Pod can starve neighbors
  • HPA needs requests to compute utilization
  • QoS class is derived from requests vs limits
spec:
  containers:
  - name: hello-world
    image: bmuschko/nodejs-hello-world:1.0.0
    resources:
      requests:
        cpu: 100m
        memory: 500Mi
        ephemeral-storage: 1Gi
      limits:
        memory: 500Mi
        ephemeral-storage: 2Gi

QoS classes: Guaranteed (requests==limits everywhere), Burstable (requests < limits), BestEffort (neither). Evicted in reverse order under pressure.

🚧 ResourceQuota & LimitRange

ResourceQuota — namespace-level cap

  • Caps total requests and limits across the namespace
  • Caps object counts: pods, services, PVCs, etc.
  • If a new Pod would exceed quota, creation is rejected
apiVersion: v1
kind: ResourceQuota
metadata:
  name: app
spec:
  hard:
    pods: "2"
    requests.cpu: "2"
    requests.memory: 500Mi

LimitRange — per-Pod/Container constraints

  • Sets min, max, and default requests/limits
  • Pods missing requests get the defaults applied
  • Pods exceeding max are rejected at admission
apiVersion: v1
kind: LimitRange
metadata:
  name: cpu-limit-range
spec:
  limits:
  - type: Container
    min: {cpu: "200m"}
    max: {cpu: "500m"}

📝 ConfigMaps

What & Why

  • Decouples non-secret configuration from images
  • Same image, different ConfigMaps → different environments
  • Consumed as: env vars, command-line args, or files in a volume
  • Updates to the ConfigMap are seen by Pods when consumed as a volume (eventually); env vars are not updated
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  application.yaml: |
    environment: prod
    feature_x: enabled
---
spec:
  containers:
  - name: backend
    image: nginx:1.23.4-alpine
    volumeMounts:
    - name: config
      mountPath: /etc/config
  volumes:
  - name: config
    configMap:
      name: app-config

🤫 Secrets

Same Shape, Different Intent

  • Like ConfigMap, but values are base64-encoded
  • At rest in etcd they may be encrypted (cluster config)
  • Mounted Secrets are tmpfs in the Pod — never written to disk
  • Built-in types: Opaque, kubernetes.io/dockerconfigjson, tls

Base64 ≠ encryption. Don't commit Secrets to git as if they were safe.

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  db-password: cGFzc3dk   # base64("passwd")
---
spec:
  containers:
  - name: backend
    image: nginx:1.23.4-alpine
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: db-password

🔒 Security Context

Pod-Level vs Container-Level

  • Some fields only valid at Pod level: fsGroup, runAsGroup
  • Some only at Container level: allowPrivilegeEscalation, capabilities, readOnlyRootFilesystem
  • Container-level overrides Pod-level when both set

The Big Knobs

  • runAsUser / runAsGroup — drop root
  • runAsNonRoot: true — refuse to start as root
  • readOnlyRootFilesystem — write only to mounted volumes
  • allowPrivilegeEscalation: false
apiVersion: v1
kind: Pod
metadata:
  name: busybox-security-context
spec:
  securityContext:
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
  containers:
  - name: busybox
    image: busybox:1.36
    command: ["sh","-c","sleep 1h"]
    volumeMounts:
    - name: vol
      mountPath: /data/test
    securityContext:
      allowPrivilegeEscalation: false
  volumes:
  - name: vol
    emptyDir: {}
