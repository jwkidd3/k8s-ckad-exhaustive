📦 Containers: A Quick Refresher

What's in a Container Image?

  • A layered filesystem
  • Default command (CMD) and entrypoint (ENTRYPOINT)
  • Declared ports, env vars, user, working dir
  • Metadata: labels, exposed volumes

Dockerfile Building Blocks

FROM node:20.1.0-alpine
WORKDIR /app
COPY package.json .
RUN npm install
COPY index.js .
EXPOSE 3000
CMD ["node", "index.js"]

Build, Run, Inspect, Save

# Build with a versioned tag
docker build -t nodejs-hello-world:1.0.0 .

# Run, mapping host 80 → container 3000
docker run -d -p 80:3000 nodejs-hello-world:1.0.0

# Hit the app
curl localhost

# Save the image to a tarball
docker save -o nodejs-hello-world-1.0.0.tar \
  nodejs-hello-world:1.0.0

Tip: CKAD images should always be pinned by tag — never :latest.

🧩 Pods: The Smallest Deployable Unit

What is a Pod?

  • One or more containers that share a network namespace and storage
  • The atomic unit of scheduling — kube-scheduler places a Pod on a node, not individual containers
  • Pods are mortal — they aren't healed if deleted directly (that's a Deployment's job)

Minimal Pod Manifest

apiVersion: v1
kind: Pod
metadata:
  name: mypod
spec:
  containers:
  - name: nginx
    image: nginx:1.23.4
    ports:
    - containerPort: 80

Speedy Creation

# Imperative create — fastest path
kubectl run mypod --image=nginx:1.23.4

# Imperative with port + dry-run for a YAML stub
kubectl run mypod --image=nginx:1.23.4 \
  --port=80 --dry-run=client -o yaml > pod.yaml

# Inspect status
kubectl get pods
kubectl describe pod mypod
kubectl logs mypod

On the exam: --dry-run=client -o yaml is your friend. Memorize it.

🔍 Pod Lifecycle & Common Failure Modes

Phases You Will See

  • Pending — waiting on scheduling or image pull
  • Running — at least one container is up
  • Succeeded — all containers exited 0 (Jobs)
  • Failed — at least one container exited non-zero
  • CrashLoopBackOff — repeated exits, kubelet is backing off

When Things Break

SymptomFirst Check
ImagePullBackOffImage name & tag, registry auth
CrashLoopBackOffkubectl logs -p (previous)
Pendingkubectl describe events
OOMKilledContainer memory limit too low

⚙️ Run-to-Completion: Jobs

When to Use a Job

  • Batch processing
  • Database migrations
  • One-off cleanup tasks
  • Anything that should run, succeed, and stop

Key Knobs

  • completions — how many successful pods needed
  • parallelism — how many pods to run at once
  • backoffLimit — retries before giving up
  • restartPolicy: Never | OnFailure

Example

apiVersion: batch/v1
kind: Job
metadata:
  name: random-hash
spec:
  parallelism: 2
  completions: 5
  backoffLimit: 4
  template:
    spec:
      containers:
      - name: random-hash
        image: alpine:3.17.3
        command: ["/bin/sh", "-c",
          "echo $RANDOM | base64 | head -c 20"]
      restartPolicy: Never

⏰ Scheduled Work: CronJobs

CronJob = Job + Schedule

  • Standard cron syntax: * * * * *
  • Each run creates a new Job, which creates Pods
  • concurrencyPolicy handles overlap: Allow / Forbid / Replace
  • successfulJobsHistoryLimit trims old Jobs

Gotcha

Don't set jobTemplate.metadata.name. The controller auto-generates a unique name per run; a static name causes "already exists" failures on every subsequent fire.

apiVersion: batch/v1
kind: CronJob
metadata:
  name: current-date
spec:
  schedule: '* * * * *'
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: Never
          containers:
          - name: current-date
            image: nginx:1.23.4
            command:
            - /bin/sh
            - -c
            - 'echo "Current date: $(date)"'

💾 Volumes: Ephemeral vs Persistent

emptyDir — Scratch Space

  • Lives as long as the Pod
  • Cleared when the Pod is deleted
  • Perfect for inter-container sharing within a Pod
  • No declaration of storage class needed
spec:
  containers:
  - name: web
    image: nginx:1.23.4
    volumeMounts:
    - name: scratch
      mountPath: /var/cache
  volumes:
  - name: scratch
    emptyDir: {}

PersistentVolume + Claim — Durable

  • PV is the actual storage (NFS, hostPath, cloud disk)
  • PVC is a request bound to a PV
  • The Pod mounts the PVC, not the PV directly
  • Decoupling means a Pod can be re-scheduled and reconnect to the same data

Static vs Dynamic provisioning: Static = admin pre-creates the PV. Dynamic = a StorageClass provisions it on PVC request.

🚦 Init Containers

Why?

  • Run setup work before the main container starts
  • Each init container runs to completion sequentially
  • If any init fails, the Pod is restarted (per restartPolicy)
  • Common uses: fetch config, wait for a service, prep a volume
spec:
  initContainers:
  - name: configurer
    image: busybox:1.36
    command: ['sh','-c','wget -O /usr/shared/app/config.json https://...']
    volumeMounts:
    - name: configdir
      mountPath: /usr/shared/app
  containers:
  - name: web
    image: bmuschko/nodejs-read-config:1.0.0
    volumeMounts:
    - name: configdir
      mountPath: /usr/shared/app
  volumes:
  - name: configdir
    emptyDir: {}

🧱 Multi-Container Patterns

Sidecar / Adapter

  • Second container enhances the main one
  • Adapter: transforms output into a target format (e.g. legacy app → Prometheus metrics)
  • Communication via shared volume or localhost

Ambassador

  • Second container proxies external traffic for the main one
  • Main container talks to localhost; ambassador handles auth, retries, rate-limiting
  • Move complexity out of the app, into a reusable proxy

Adapter Shape

┌──── Pod ──────────────┐ │ ┌───────┐ ┌────────┐ │ │ │ app │→ │adapter │→│ metrics │ └───┬───┘ └────────┘ │ │ ▼ shared volume │ └───────────────────────┘

Ambassador Shape

┌──── Pod ──────────────┐ │ ┌───────┐ ┌─────────┐│ │ │ app │→ │ambassad.│┼→ external │ └───────┘ └─────────┘│ │ localhost:8080 │ └───────────────────────┘

⎈ Packaging with Helm

What Helm Adds

  • A package manager for Kubernetes
  • Charts bundle templated manifests + default values
  • Installing a chart = templates rendered with values → kubectl apply
  • Lets you ship a complex stack (e.g. Prometheus) with one command

Core Commands

helm repo add prometheus-community \
  https://prometheus-community.github.io/helm-charts
helm repo update
helm search hub kube-prometheus-stack
helm install <name> <chart>
helm list
helm uninstall <name>

Anatomy of a Chart

my-chart/ ├── Chart.yaml ← name, version, apiVersion ├── values.yaml ← default values ├── templates/ │ ├── deployment.yaml │ └── service.yaml └── charts/ ← subcharts
